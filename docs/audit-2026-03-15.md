## S. Security

### S1. SQL injection via lstr/rstr in api.sql()

**Severity:** CRITICAL
**File:** `api/api.sql:192-257`

**Problem:** The `api.sql()` query builder accepts `lstr` and `rstr` fields from user-supplied JSON search parameters. These values are concatenated directly into the WHERE clause without any sanitization:

```sql
vLStr := coalesce(r.lstr, '');
vRStr := coalesce(r.rstr, '');
-- ...
vWhere := E'\n WHERE ' || vLStr || vField || GetCompare(vCompare) || vValue || vRStr;
```

While `vField` is validated against `arColumns` and `vValue` goes through `quote_nullable()`, `vLStr`/`vRStr` are raw user input. A payload like `{"lstr": "1=1 UNION SELECT secret FROM oauth2.audience --"}` injects verbatim.

This function is called from every `api.list_*` endpoint (~67 files), making it a system-wide injection surface.

**Decision:** `[x] Approved`
**Notes:** Fixed: whitelist `(` and `)` only via CASE expression.

### S2. ORDER BY without validation

**Severity:** HIGH
**File:** `api/api.sql:280-284`

**Problem:** The ORDER BY column validation is commented out:

```sql
--PERFORM CheckJsonbValues('orderby', array_cat(arColumns, array_add_text(arColumns, ' desc')), pOrderBy);
IF jsonb_typeof(pOrderBy) = 'array' THEN
  IF JsonbToStrArray(pOrderBy) IS NOT NULL THEN
    vSelect := vSelect || E'\n ORDER BY ' || array_to_string(array_quote_literal_json(JsonbToStrArray(pOrderBy)), ',');
  END IF;
END IF;
```

`array_quote_literal_json` does NOT validate that values are actual column names. Arbitrary expressions in the `orderby` array are interpolated into ORDER BY. Affects every list endpoint.

**Decision:** `[x] Approved`
**Notes:** Fixed: new JsonbToOrderBy() function with whitelist + safe function-call auto-allow via full-match regex. Zero config-layer changes.

### S3. Aggregate regex bypass in JsonbToFields()

**Severity:** HIGH
**File:** `kernel/general.sql:1296-1298`

**Problem:** The regex auto-allows aggregate expressions but uses partial match:

```sql
IF regexp_like(r.value, '\m(sum|count|max|min|avg)\s*\(\s*(?:DISTINCT\s+)?...', 'i') THEN
  pFields := array_append(pFields, r.value);
END IF;
```

A payload like `count(*), (SELECT usename FROM pg_catalog.pg_shadow LIMIT 1)` passes the regex (contains `count(*)`) and gets appended to the allowed fields list, enabling SELECT-clause injection.

**Decision:** `[x] Approved`
**Notes:** Fixed: partial-match regex replaced with full-match `^...$` plus optional `AS alias` support.

### S4. Unrestricted endpoint registration enables code execution

**Severity:** HIGH
**File:** `api/routine.sql:126-165`, `api/api.sql:404`

**Problem:** `api.run()` executes arbitrary SQL stored in `db.endpoint.definition`:

```sql
FOR r IN EXECUTE GetEndpointDefinition(uEndpoint) USING pPath, pPayload
```

`AddEndPoint()`/`EditEndPoint()` have no authorization checks — any function that calls them can register arbitrary SQL that executes as `kernel`.

**Decision:** `[x] Approved`
**Notes:** Added format validation in AddEndPoint — only canonical `SELECT * FROM rest.<name>($1, $2);` accepted.

### S5. Dynamic function call without identifier quoting in observer

**Severity:** HIGH
**File:** `observer/routine.sql:877-879`

**Problem:**

```sql
vRoutine := coalesce(GetClassCode(...), GetEntityCode(...), 'object');
FOR e IN EXECUTE format('SELECT * FROM api.get_%s($1)', vRoutine) USING ...
```

Uses `format('%s')` instead of `format('%I')`. If a class/entity code contains malicious content, it injects into the EXECUTE. Same pattern at line 897.

**Decision:** `[x] Approved`
**Notes:** Fixed: format('%s') replaced with format('%I') at lines 879 and 897.

### S6. Dynamic function name in report module

**Severity:** HIGH
**File:** `report/form/routine.sql:153`, `report/ready/routine.sql:162`

**Problem:**

```sql
EXECUTE 'SELECT report.' || GetReportFormDefinition(pForm) || '($1, $2);'
EXECUTE 'SELECT report.' || r.definition || '($1, $2);'
```

Function names from `db.report_form.definition` / `db.report_routine.definition` are concatenated without identifier escaping. Should use `format('%I', ...)` or `quote_ident()`.

**Decision:** `[x] Approved`
**Notes:** Fixed: string concatenation replaced with format('%I', ...) in both BuildReportForm and ExecuteReportReady.

### S7. OAuth2 client secret stored in plain text

**Severity:** HIGH
**File:** `oauth2/table.sql:107`, `oauth2/routine.sql:386-387`

**Problem:** `oauth2.audience` stores the client secret in **both** plain text and hash:

```sql
CREATE TABLE oauth2.audience (
    secret      text NOT NULL,   -- plain text
    hash        text NOT NULL,   -- crypt() hash
);
```

The plain-text `secret` column is used for JWT signing. Any SQL injection that reads `oauth2.audience` exposes all client secrets.

**Decision:** `[x] Won't fix`
**Notes:** This is a JWT signing key, not a password. HMAC-based JWT requires the original key in plaintext — cannot be hashed. Real protection is preventing table reads (S1-S3 closed).

### S8. JWT sign/verify GRANT TO PUBLIC

**Severity:** HIGH
**File:** `kernel/jwt.sql:19-122`

**Problem:** All JWT functions are granted to PUBLIC:

```sql
GRANT EXECUTE ON FUNCTION sign(json, text, text) TO PUBLIC;
GRANT EXECUTE ON FUNCTION verify(text, text, text) TO PUBLIC;
```

Combined with S7, any database role can forge JWTs if it obtains a client secret.

**Decision:** `[x] Approved`
**Notes:** Fixed: removed all 5 GRANT EXECUTE ... TO PUBLIC from jwt.sql. Existing DBs need REVOKE patch.

### S9. PUBLIC functions without SECURITY DEFINER

**Severity:** HIGH
**File:** `kernel/public.sql:525-683`, `kernel/jwt.sql`

**Problem:** 11+ PUBLIC-granted functions lack `SECURITY DEFINER SET search_path = ...`:

- `StrToTimeStamp`, `StrToTimeStampTZ`, `StrToTime`, `StrToInterval`
- `IntToStr`, `StrToInt`, `DateToStr` (3 overloads), `StrToDate`
- `MINDATE`, `MAXDATE`
- Multiple `CheckNull` variants
- JWT: `url_encode`, `url_decode`, `algorithm_sign`, `sign`, `verify`

Vulnerable to search_path hijacking if attacker controls public schema.

**Decision:** `[x] Approved`
**Notes:** Fixed: added `SET search_path = kernel, public, pg_temp` to 63 plpgsql functions in kernel/public.sql. 5 functions already had SET search_path — skipped.

### S10. search_path with public schema first

**Severity:** IMPORTANT
**File:** `kernel/general.sql:29`, `file/routine.sql:727`

**Problem:** Two functions have `public` first in search_path:

```sql
SET search_path = public, pg_temp;           -- gen_kernel_uuid()
SET search_path = public, kernel, pg_temp;   -- S3 signature function
```

Standard codebase pattern is `kernel, pg_temp`. Public-first means public schema objects take priority — hijacking vector.

**Decision:** `[x] Approved`
**Notes:** Fixed: both functions changed to `kernel, pg_temp` (gen_kernel_uuid in general.sql, S3 function in file/routine.sql).

### S11. gen_salt('md5') instead of gen_salt('bf')

**Severity:** IMPORTANT
**File:** `oauth2/routine.sql:387`, `admin/routine.sql:288`

**Problem:** Password/secret hashing uses MD5-crypt:

```sql
crypt(pSecret, gen_salt('md5'))
```

MD5-crypt is significantly weaker than bcrypt (`gen_salt('bf')`). Computationally cheap to brute-force.

**Decision:** `[x] Approved`
**Notes:** Fixed: gen_salt('md5') → gen_salt('bf') in oauth2/routine.sql (1), admin/routine.sql (3), admin/table.sql (2). Backward compatible — crypt() auto-detects hash format at verification.

## P. Performance

### P1. aou() full table scan on every list query

**Severity:** CRITICAL
**File:** `admin/routine.sql` (aou function)

**Problem:** The `aou()` function scans all AOU (Access Object Unit) entries on every list query. No index-driven filtering for user+entity+permission. This is the hottest path in the system — every `api.list_*` call triggers it through the Object/AccessObject views.

**Decision:** `[x] Won't fix`
**Notes:** Entity-specific Access* views (e.g. AccessJob) already inline the aou pattern with direct JOIN on db.aou + member_group, bypassing the generic aou() function. Indices (userid), (entity, userid, mask) cover the hot paths. The generic aou() is only used in AccessObject/AccessObjectId — acceptable for current scale.

---

### P2. Object view joins 14 tables, ObjectDocument joins 18+

**Severity:** CRITICAL
**File:** `entity/object/view.sql`

**Problem:** The base `Object` view joins 14 tables (object, entity, class_tree, type, state_type, state, plus _text variants, plus users). `ObjectDocument` adds area, priority, and more _text tables for 18+ JOINs. Every entity list query runs through these views, and PostgreSQL cannot always prune unused joins.

**Decision:** `[x] Won't fix`
**Notes:** Flat views are denormalized projections for REST API. PostgreSQL applies join elimination for unused LEFT JOINs. INNER JOINs required for integrity. Splitting into smaller views adds complexity without guaranteed benefit.

---

### P3. Getter functions missing STABLE marker

**Severity:** IMPORTANT
**File:** Various modules

**Problem:** ~15 getter functions (read-only, no side effects) are declared as default `VOLATILE` instead of `STABLE`. PostgreSQL cannot optimize repeated calls within a single query. Examples include `GetEntity()`, `GetClass()`, `GetState()`, and similar lookup functions.

**Decision:** `[x] Approved`
**Notes:** Fixed: +128 SQL functions marked STABLE STRICT (or STABLE for functions with DEFAULT params). 1 skipped: ClearMethodStack (calls INSERT). 28 files across 22 modules.

---

### P4. current_locale() redundant PK lookups

**Severity:** IMPORTANT
**File:** `current/routine.sql`

**Problem:** `current_locale()` calls `current_session()` which does a PK lookup on `db.session`, then `current_locale()` does another PK lookup on `db.session` for the locale field. Two round-trips to the same row.

**Decision:** `[x] Won't fix`
**Notes:** current_locale() already marked STABLE — PostgreSQL caches result within a single query. Double PK lookup happens at most once per statement, not per row.

---

### P5. N+1 in ExecuteMethodForAllChild() and CopyClass()

**Severity:** IMPORTANT
**File:** `workflow/routine.sql`

**Problem:** These functions iterate over child records and execute a separate query per child (classic N+1 pattern). With large hierarchies, this degrades significantly.

**Decision:** `[x] Won't fix`
**Notes:** ExecuteMethodForAllChild — side-effectful workflow transitions, cannot be batched. CopyClass — rare admin operation, N+1 is non-critical.

## A. Architecture

### A1. REST dispatcher cloning (~5800 lines)

**Severity:** HIGH
**File:** 34 `rest.sql` files across all modules

**Problem:** All entity REST dispatchers are structurally identical (~190 lines each). The only difference is the entity name. Additionally, each dispatcher duplicates an array/record if-then-else pattern (~195 instances total) for handling both JSONB array and single-record payloads.

The platform already has metadata infrastructure (`GetRoutines()`, `GetEntity()`, `ExecuteDynamicMethod()`) that could drive a single parameterized dispatcher.

**Estimated waste:** ~5800 lines of copy-paste.

**Decision:** `[x] Deferred`
**Notes:** v2.0 — single parameterized dispatcher using existing metadata infrastructure. Too large for current cycle; requires backward compatibility with configuration-layer rest.sql files.

### A5. Dual error system (exception + error_catalog)

**Severity:** MEDIUM
**File:** `exception/routine.sql`, `error/init.sql`, `docs/plans/2026-03-15-exception-localization-design.md`

- **New:** `db.error_catalog` + `db.error_catalog_text` with 480 `RegisterError()` entries and full REST API

Migration plan exists (14 steps), steps 1-5 appear complete, steps 6-14 pending.

**Decision:** `[x] Deferred`
**Notes:** v2.0 — migrate all exception calls to RegisterError/ParseMessage unified system.

## D. Developer Experience

### D1. Fragile UUID detection pattern duplicated 9 times

**Severity:** MEDIUM
**File:** `api.sql` and `routine.sql` across 9 modules

**Problem:** The UUID detection heuristic:

```sql
IF length(pCode) = 36 AND SubStr(pCode, 15, 1) = '4' THEN
    RETURN pCode;
END IF;
```

is duplicated in 9 files. It checks only UUID v4 (position 15 = '4') and fails for v1, v5, v6, v7. A centralized `IsUUID(text)` function would be more robust and DRY.

**Decision:** `[x] Won't fix`
**Notes:** Current pattern works for v4 UUIDs generated by gen_kernel_uuid(). Platform never produces other UUID versions.

---

### D2. Dual error patterns (localized vs hardcoded RAISE)

**Severity:** MEDIUM
**File:** Mixed across REST and utility modules

**Problem:** Two competing error patterns:

- **Pattern A** (509 uses): `PERFORM AccessDenied();` — localized via exception module
- **Pattern B** (116 uses): `RAISE EXCEPTION 'ERR-40000: ...'` — hardcoded, bypasses localization

Pattern B is concentrated in utility functions (`file/routine.sql`, `observer/routine.sql`). Some Pattern B messages are in Russian.

**Decision:** `[x] Deferred`
**Notes:** v2.0 — part of A5 unified error_catalog migration.

---

### D3. Only 1 RAISE NOTICE in 800+ functions

**Severity:** MEDIUM
**File:** Entire codebase

**Problem:** Virtually no debug logging in the entire framework. Makes troubleshooting production issues extremely difficult without external tooling.

**Decision:** `[x] Won't fix`
**Notes:** Debug logging via RAISE NOTICE creates noise in production. Current approach: WriteToEventLog() for audit trail, PG_EXCEPTION_CONTEXT in api.run() for errors.

---

### D4. Wiki teaches api.count_* pattern no entity uses

**Severity:** LOW
**File:** `wiki/` documentation

**Problem:** The wiki tutorial documents an `api.count_*` pattern for entity counting, but no real entity in the platform implements this. All counting goes through `api.list_*` with `count(*)`. Documentation misleads developers.

**Decision:** `[x] Approved`
